Haskell-based Virtual Machine

For Execution of ARM Assembly Code

Introduce self and project.
  • Small core
  • Leverage functional paradigm

Motivation

Used ARM7 in first year, and in second year. In third year wrote a compiler for an entirely made up assembly language. Seemed wasteful.

Design

Mirroring Reality

Organised similarly to a true machine - CPU, ALU, register file, etc.

Von Neumann Architecture

True Von Neumann machine with single memory for data and instructions

Leveraging Functional Power

  • GADTs used heavily to ensure strict type safety
  • Laziness allows 4GiB memory space to be provided but only allocated on demand
  • Record types were useful but proved limiting. Would've liked to convert to Lenses but needed time to grok

Small monadic core

Only monadic components are CPU and instruction decoder: State and Get monads

Implementation

Instructions

Conditional execution. Dataflow operations.

ALU

Pure - given instruction and some context returns new context. Executes arithmetic, logical and shift operations.

CPU

State monad where the state is a "Machine" datatype. Handles branching and stateful operations, such as providing context to ALU and updating context after instructions. Instructions pipelined into Fetch - Decode - Execute stages, and pipeline can be stalled by memory operations or flushed by branch operations

Memory

Full 4GiB addressable space as previously mentioned. Implemented as an IntMap (internally a lazy radix trie/patricia trie) of Word32 addresses to Word8 values

Decoder

Interesting section. Implemented using Data.Binary, takes 32 bit instruction word and reconstitutes Instruction GADT. Monadic with Data.Binary.Get, and parsing simplified using Control.Applicative

What's left?

  • Front end parser

  • 16-bit load/stores

  • Who knows?

Unfortunately uncertain as to which features will be implemented by deadline and which will be "further work"

Global State

Set data-state="something" on a slide and "something" will be added as a class to the document element when the slide is open. This lets you apply broader style changes, like switching the background.

Custom Events

Additionally custom events can be triggered on a per slide basis by binding to the data-state name. 


Reveal.addEventListener( 'customevent', function() {
    console.log( '"customevent" has fired' );
} );
